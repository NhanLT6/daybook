How should log items be editable?

Both options show AI results as chat messages with a Save button. The difference is whether you can tweak individual logs inline.

A

Inline edit per log item

Fine-grained control
DS-1234 for 2h, DS-5678 for 1h today
I found 2 logs. Edit any field before saving:
Project DS-1234
Task Development
Duration 120 min
Date 2026-04-15
Project DS-5678
Task Development
Duration 60 min
Date 2026-04-15
↑ click any field to edit

Each log returned by AI is an editable card. Fields (project, task, duration, date) can be clicked and modified directly. User can also delete individual items before saving.

Pros

  • Fix one wrong field without re-prompting AI
  • Remove a bad log without losing the others
  • Fastest path for minor corrections

Cons

  • More complex UI to build
  • Cards take more vertical space in chat
  • Two ways to edit (inline + follow-up chat)
B

Read-only + fix via follow-up

Simpler, chat-first
DS-1234 for 2h, DS-5678 for 1h today
I found 2 logs — does this look right?
DS-1234 · Development · 2026-04-15
120 min
DS-5678 · Development · 2026-04-15
60 min
"DS-5678 was actually 1.5h, not 1h"
Updated! Ready to save?
DS-1234 · Development · 2026-04-15
120 min
DS-5678 · Development · 2026-04-15
90 min
✓ 2 logs saved

Log items are displayed read-only in the chat message. To fix anything, user types a follow-up — AI updates and re-presents. The updated item is highlighted in the new message. Save button appears on each AI response that contains logs.

Pros

  • Simpler UI — no editable fields in chat
  • Pure chat pattern, consistent UX
  • AI can handle complex corrections naturally
  • Conversation history shows all changes

Cons

  • Tiny fix = extra round-trip to AI
  • Costs an API call for each correction
  • Slower if many small corrections needed
